# Codex Config Switcher 产品与体验优化方案

日期：2026-04-09

## 1. 背景与分析口径

当前版本 `0.1.0` 已经具备核心能力：读取当前 Codex 配置、保存多个预设、快速应用、自动备份，以及应用后重启目标软件。从产品经理和用户视角看，这个产品已经完成了“能用”的第一步，但距离“敢用、爱用、持续高频使用”还差一层体验设计。

这份方案基于当前实现做分析，重点参考了以下界面与交互入口：

- 菜单栏快速切换入口：`Sources/CodexConfigSwitcher/Views/MenuBarContentView.swift`
- 主窗口总布局：`Sources/CodexConfigSwitcher/Views/MainWindowView.swift`
- 预设编辑主表单：`Sources/CodexConfigSwitcher/Views/PresetEditorView.swift`
- 核心状态与操作流程：`Sources/CodexConfigSwitcher/AppModel.swift`

分析时采用两个视角：

- 产品经理视角：这个工具是否具备清晰定位、合理信息架构、明确优先级和后续演进空间。
- 用户视角：第一次打开是否看得懂，高频切换是否足够快，关键操作是否足够安全，出错时是否容易恢复。

我这里先做一个合理假设：当前核心用户不是纯开发新手，而是“会改 Codex 配置，但不想反复手改文件”的效率型用户。这类用户愿意使用工具，但不愿意承担配置被误改、字段看不懂、切换后不确定有没有生效的风险。

## 2. 当前产品的优点

先明确现阶段已经做对的事情，这些应该保留：

- 产品定位明确：不是通用配置中心，而是围绕 Codex 本地配置切换的垂直工具。
- 用户收益直接：把“打开文件、复制粘贴、保存、重启应用”的链路，压缩成了预设保存和一键应用。
- 双入口设计是对的：菜单栏负责快切，主窗口负责编辑，这个职责拆分方向正确。
- 有备份机制，降低了误操作损失，这一点对工具信任度非常重要。
- 支持从 live 配置导入预设，避免用户第一次使用时完全从零配置。

也就是说，当前版本不是“功能缺失型产品”，而是“功能已具雏形，但信息组织和交互可信度还不够”的状态。接下来优化重点不应该是继续堆字段，而应该是降低理解成本、降低误操作风险、提高切换确定性。

## 3. 主要问题诊断

### 3.1 信息架构混杂，用户心智不清晰

当前主窗口把三类内容混在同一个长页面里：

- 预设本身的内容
- 全局设置，例如目标文件路径、目标软件路径
- 状态信息，例如最近读取、最近写入、备份位置

这会带来一个明显问题：用户分不清“我现在在编辑预设”还是“我在改应用全局设置”。例如文件路径和重启目标应用属于低频全局配置，但现在被放在编辑流程靠前的位置，容易打断主要任务。对于大部分用户来说，真正高频的是：

1. 选一个预设
2. 检查几个关键字段
3. 应用

而不是每次都先看到路径、Bundle ID、App 路径这些偏底层信息。

### 3.2 表单过于技术化，理解门槛偏高

当前表单大量直接暴露原始配置键，例如：

- `model_provider`
- `review_model`
- `wire_api`
- `windows_wsl_setup_acknowledged`

这对熟悉 Codex 配置文件的人没问题，但对“知道自己要切换 API 地址和模型，却不熟悉完整字段语义”的用户来说，理解成本很高。尤其是部分低频字段和高频字段被放在同一层级，会让用户无法快速判断“哪些必须改，哪些可以不动”。

从产品视角看，这会削弱工具的普适性；从用户视角看，这会造成“我怕改错，所以宁可回去手改文件”的心理。

### 3.3 关键状态没有被清楚区分

当前界面里至少有三种状态，但没有被明确表达：

- 当前选中的预设
- 当前编辑中的草稿
- 当前 live 生效的配置

这三者在概念上并不相同，但 UI 中没有强提醒区分。比如菜单栏高亮使用的是 `selectedPresetID`，它更接近“选中项”，不一定等于“当前真实生效项”。如果用户在外部改过配置，或者切换后没有及时重新感知 live 状态，就容易产生“界面显示这个预设，但系统实际是不是这个配置”的疑问。

这是一个非常关键的体验问题，因为配置类工具的核心价值是“可控”和“确定”。

### 3.4 高频操作路径还不够短

菜单栏入口目前已经有快速应用能力，但还没有把高频路径压缩到极致，主要问题有：

- 没有清楚标识“当前生效预设”
- 没有收藏、最近使用、环境标签等高频识别辅助
- 快速应用后反馈较弱，用户需要自己读状态文本确认
- 次级操作和高频操作没有很好分层

用户真正希望的是：看一眼就知道自己现在在哪个环境，点一下就切过去，并且立刻知道切换是否成功。

### 3.5 安全感和信任感还不够完整

虽然当前版本已经有备份机制，但从用户视角看，信任链路还缺几环：

- 应用前没有明显校验或预检
- 没有“将发生什么变化”的差异预览
- 删除预设没有更明确的风险控制
- API Key 作为敏感信息，当前产品层面还没有更强的安全心智表达
- 备份可以打开目录，但还没有形成“快速恢复”的闭环

工具类产品一旦触碰配置写入和密钥管理，用户对“安全感”的要求会明显高于一般表单产品。

## 4. 优化目标

建议把下一个阶段的产品目标定义为：

“把 Codex Config Switcher 从一个可编辑配置文件的小工具，升级为一个可理解、可验证、可恢复的配置切换工作台。”

围绕这个目标，建议关注四个体验指标：

- 更快：用户从打开应用到完成一次切换，尽量控制在 10 秒内。
- 更懂：用户第一次使用时，能看懂哪些是常用项，哪些是高级项。
- 更稳：用户在应用前知道会改什么，应用后知道有没有生效，出错后知道怎么恢复。
- 更省心：全局设置不干扰高频切换，菜单栏可以承担绝大多数日常操作。

## 5. 方案设计

### 5.1 重新梳理信息架构

建议把产品结构改成“三层架构”：

- 第一层：菜单栏快切层
  目标是“2 次点击内完成切换”。
- 第二层：主窗口工作台
  目标是“管理预设、编辑关键字段、查看当前状态”。
- 第三层：高级设置层
  目标是承载路径、目标应用、备份策略、低频高级参数。

这意味着主窗口不应该再是一页长表单，而应该变成“管理台 + 编辑台”的组合。推荐结构如下：

- 左侧：预设列表
- 右侧上方：当前状态摘要
- 右侧中部：基础配置编辑区
- 右侧下方或折叠区：高级参数
- 全局设置单独放到“设置”弹窗或独立标签页

这个结构的好处是，用户一进来就能先看到“当前在用什么”，而不是立刻面对一大堆技术字段。

### 5.2 主窗口布局优化

建议将主窗口改为以下布局：

#### 左侧预设列表

- 每个预设显示：名称、环境标签、Base URL 简称、Model
- 增加状态标识：`当前生效`、`草稿已修改`、`最近使用`
- 支持右键或更多菜单：复制、重命名、删除、导出
- 列表头部增加搜索和“新建预设”按钮

这样可以把“管理预设”的效率提升上来，也能降低预设越来越多后的混乱感。

#### 右侧摘要区

页面顶部增加一块“当前状态摘要卡片”，集中展示：

- 当前 live 配置对应的环境
- 最近应用时间
- 目标应用状态：运行中 / 未运行 / 未安装
- 当前是否存在未保存修改

这块区域的作用，是把用户最关心的“现在到底是什么状态”放到最高优先级位置。

#### 右侧编辑区

默认只展示最常用字段：

- 预设名称
- 环境标签
- Base URL
- Model
- Review Model
- API Key
- 认证模式
- 应用按钮
- 测试连接按钮

其余字段归入“高级配置”折叠面板，默认收起。这样既保留能力，又不会让页面一上来太重。

### 5.3 菜单栏布局优化

菜单栏应该更像“切换器”，而不是缩小版主窗口。建议结构调整为：

- 顶部：当前生效环境卡片
- 中部第一组：收藏 / 最近使用预设
- 中部第二组：全部预设
- 底部第一组：重新读取、打开主窗口、恢复最近备份
- 底部第二组：退出

每个预设项建议增加：

- 环境标签
- 当前生效对勾或高亮
- 一行简短描述，例如 `official / gpt-5.4`

应用成功后建议在菜单栏内立即出现明确反馈，例如：

- `已切换到：生产代理`
- `Codex 即将重启`
- `切换成功，可直接继续使用`

这样用户就不需要靠猜测去确认操作结果。

### 5.4 交互逻辑优化

这是本轮最重要的部分，建议重点优化以下链路。

#### 首次使用流程

第一次打开时，不建议直接把完整表单丢给用户。推荐改成引导式流程：

1. 自动检测 `config.toml` 和 `auth.json`
2. 读取当前 live 配置
3. 询问用户是否“导入为默认预设”
4. 提示“以后可以在菜单栏快速切换”

这样能快速建立正确心智：这个工具不是让你从零写配置，而是帮助你基于现有配置做切换和管理。

#### 编辑与保存流程

当前产品建议增加“脏状态”识别：

- 用户改过但未保存时，显示 `未保存`
- 切换预设或关闭窗口前，提示是否保存
- 对“覆盖当前预设”和“另存为新预设”做更明确区分

这能大幅减少“我明明改了，为什么下次没了”的困惑。

#### 应用流程

推荐把“立即应用”改造成四步流程：

1. 本地校验
2. 差异预览
3. 正式写入
4. 成功反馈与后续动作

其中本地校验至少包括：

- Base URL 是否为合法 URL
- 必填字段是否为空
- API Key 是否缺失
- `model_context_window` 等数值是否合理

差异预览不需要很复杂，哪怕只显示“本次将变更的字段列表”，也能显著提升可控感。

#### 删除与恢复流程

删除预设建议补两层保护：

- 二次确认
- 短时撤销

另外，备份机制已经存在，下一步应该把“恢复最近一次备份”做成明确动作，而不是只打开备份目录让用户自己找。

### 5.5 文案与字段表达优化

当前很多字段名称更像开发者内部名，而不是用户可理解文案。建议采用“双层表达”：

- 主标题用用户语言
- 辅助说明里展示底层键名

例如：

- `接口地址`
  辅助说明：`base_url`
- `主模型`
  辅助说明：`model`
- `评审模型`
  辅助说明：`review_model`

像 `windows_wsl_setup_acknowledged` 这类极低频且不适合普通用户直接编辑的字段，更适合放到高级模式中，并补一句“默认沿用当前 live 配置”。

### 5.6 安全与信任设计

建议把“安全感”当成一个独立优化主题，而不是顺带处理：

- API Key 默认掩码显示，只在需要时查看
- 预设与密钥分离存储，优先接入 macOS Keychain
- 应用前显示将修改哪些关键字段
- 应用后展示备份已生成，并可一键恢复
- 对生产环境或高风险环境增加颜色标识和二次确认

如果后续预设会越来越多，建议支持为预设增加环境类型：

- `官方`
- `代理`
- `测试`
- `备用`

这样不仅更容易管理，也更能减少切错环境的风险。

## 6. 优先级建议

### P0：应该优先做

这些改动最直接影响产品可用性和用户信任，建议优先：

- 主窗口重构为“预设管理 + 当前状态 + 基础编辑 + 高级折叠”
- 把文件路径、目标应用等全局设置移出主编辑流
- 增加 `当前生效`、`草稿未保存`、`最近应用` 状态标识
- 增加应用前校验
- 增加删除确认
- 菜单栏明确显示当前生效预设
- 为路径和应用选择增加系统选择器，减少手填路径

### P1：第二阶段做

- 增加测试连接
- 增加差异预览
- 增加收藏、最近使用预设
- 增加一键恢复最近备份
- 增加预设复制、导入、导出
- 增加基础模式 / 高级模式切换

### P2：锦上添花

- 键盘快捷键
- 环境颜色和标签体系
- 更丰富的预设排序和筛选
- 更完整的操作历史
- 团队共享预设或模板库

## 7. 建议的下一步执行顺序

如果要把方案继续落成实现，建议按下面顺序推进：

1. 先改信息架构和布局，不急着新增很多能力。
2. 再补关键状态表达和脏状态管理。
3. 然后补校验、差异预览、恢复闭环。
4. 最后再做收藏、筛选、导入导出等效率增强项。

原因很简单：当前版本最需要解决的不是“按钮不够多”，而是“用户是否足够理解当前状态、是否足够放心点击应用”。

## 8. 一句话结论

这个产品的下一阶段重点，不是继续扩展配置字段，而是把它从“能编辑配置的工具”升级为“能安全切换配置的工作台”。只要把状态表达、布局分层、应用前校验和恢复闭环做好，整体产品感受会明显上一个台阶。
